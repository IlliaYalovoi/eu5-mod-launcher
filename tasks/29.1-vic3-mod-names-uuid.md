# 29.1 — Vic3 Mod Names Display as UUIDs in Load Order

## Context

Victoria 3 mods loaded from SQLite playsets show their UUIDs (e.g. `e058682b-7509-4448-940a-ee31a96c3d33`) as names in the load order panel, even though the constraint modal correctly shows human-readable names.

## Expected Behavior

Load order should display the same mod names visible in the mod list and constraint popup.

## Root Cause

The constraint modal correctly resolves mod IDs to names because it uses `modsStore.allMods` which contains fully populated `mods.Mod` structs with `Name` field populated from `.metadata/metadata.json`. However, the load order panel relies on a `modsByID` computed that builds from `allMods.value`, but the display uses the mod's `Name` field directly.

The Vic3 SQLite schema uses mod IDs that are not the directory names — they are UUIDs assigned by Paradox Launcher. The Vic3 mod discovery (via `mods.ScanDirs`) uses directory names as IDs, but the `modPathByID` map from `ImportModList` maps UUIDs to dirPaths, not directory names to UUIDs.

## What to Fix

1. **Option A**: Make Vic3's `LoadState` return `modPathByID` keyed by the same ID used in `orderedIDs`, and ensure `Discover` can resolve names for these UUIDs.

2. **Option B**: Change the load order display to always look up the mod by ID in `allMods` (which has the correct Name) rather than relying on `modPathByID`.

3. **Verify**: Why does the constraint modal show correct names? It must be resolving IDs to mods differently — trace the data flow.
