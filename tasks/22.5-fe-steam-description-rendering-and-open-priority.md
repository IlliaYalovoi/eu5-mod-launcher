# Task 22.5 — Steam Description Rendering + Workshop Open Priority

## Goal

Improve Steam mod details UX by:

1. Rendering Steam description with Steam-like formatting (BBCode, links, tables, inline images).
2. Opening workshop links with strict fallback priority:
   - first: Steam client
   - second: OS default browser
   - third: in-app/webview fallback

## Context

Depends on: tasks 20-22.
Builds on metadata cache from task 21 and details panel from task 22.

## Deliverables

### Description rendering

- Add description rendering pipeline for `WorkshopItem.description`.
- Support Steam-common BBCode tags used in workshop pages:
  - `[b]`, `[i]`, `[u]`, `[s]`
  - `[url]...[/url]` and `[url=...]...[/url]`
  - `[img]...[/img]`
  - `[table]`, `[tr]`, `[th]`, `[td]`
  - line breaks and paragraph spacing
- Render as sanitized HTML in `frontend/src/components/ModDetailsPanel.vue`.
- Keep unsupported tags non-breaking (strip or downgrade safely).

### Embedded image association + caching

- Extract all image URLs from rendered description content.
- Cache description images by workshop item ID namespace (separate from preview thumbnail cache), e.g.:
  - cache key includes `itemId` + stable hash of image URL
  - persisted under app cache dir in a dedicated folder
- Replace remote URLs in rendered HTML with local cached file paths when available.
- Download failures must be non-blocking: show text/link content even if image cache misses.
- Add eviction/cleanup behavior aligned with metadata cache lifecycle.

### Workshop open priority flow

Add backend method for controlled workshop opening behavior:

```go
func (a *App) OpenWorkshopItem(itemID string) error
```

Behavior order:

1. Try Steam client deeplink (`steam://url/CommunityFilePage/<itemID>` or equivalent).
2. If Steam open fails/unavailable, open HTTPS workshop URL in OS default browser.
3. If both fail, open in app fallback webview/window.

Frontend updates:

- Replace direct anchor behavior for `Open in Steam Workshop` with backend action call.
- Show inline error only when all three strategies fail.

### Tests

- BBCode parsing/render tests for representative workshop snippets.
- Sanitization tests (no script/event-handler injection).
- Description image extraction and cache mapping tests.
- Cache read/write/eviction tests for description images.
- Open priority tests for success/failure combinations (steam success, browser fallback, webview fallback, full failure).

## Acceptance criteria

- Workshop descriptions preserve expected formatting for common Steam BBCode content.
- Embedded description images appear in details panel and are served from cache after first successful load.
- Reopening the same mod does not redownload already cached description images unless cache is stale/evicted.
- `Open in Steam Workshop` follows Steam client -> browser -> in-app fallback priority.
- UI remains responsive; metadata/image failures are surfaced without breaking mod details panel.

## Notes for agent

- Keep renderer secure by sanitizing generated HTML before mount.
- Avoid blocking the whole panel on image caching.
- Keep implementation split: parser/normalizer, cache service, UI renderer, open-link action.
- Do not regress existing thumbnail cache behavior from task 21.
